
# Классы в Swift. Отличия от структур

## Введение

В `Swift` основные строительные блоки для создания типов данных — это **классы** и **структуры**. 

Несмотря на внешнее сходство, у них есть фундаментальные различия, которые влияют на то, как они работают в памяти и как их следует использовать. 

Понимание этих различий — ключ к написанию эффективного и предсказуемого кода.


## Общие черты классов и структур

Прежде чем углубляться в различия, посмотрим, что у них общего. 

И классы, и структуры **предоставляют схожий интерфейс для определения новых типов данных.**

| Общая черта | Описание |
|-------------|----------|
| **Свойства** | Могут хранить значения (`переменные` и `константы`) |
| **Методы** | Могут содержать `функции`, которые работают с этими свойствами |
| **Инициализаторы** | Могут иметь специальные методы `init` для настройки начального состояния |
| **Расширения** | Можно расширять функциональность с помощью `extension` |
| **Протоколы** | Могут соответствовать `протоколам`, реализуя их требования |

Пример определения класса и структуры выглядит схоже:

```swift
/// Класс
class PersonClass {
    var name: String
    var age: Int
    
    /// Обазательно указать инициализатор
    init(name: String, age: Int) {
        self.name = name
        self.age = age
    }
    
    func greet() {
        print("Привет, меня зовут \(name)")
    }
}

/// Структура
struct PersonStruct {
    var name: String
    var age: Int
    
    /// Стандартный инициализатор создан автоматически, можно не указывать
    
    func greet() {
        print("Привет, меня зовут \(name)")
    }
}
```

## Главные отличия классов от структур

Теперь рассмотрим ключевые различия, которые определяют поведение этих типов.

### 1. Ссылочный тип (`Классы`) против типа-значение (`Структуры`)

Давайте рассмотрим **фундаментальное различие**. 

**Пример с классом:**

```swift
class Car {
	/// Модель автомобиля 
    var model: String
    
    /// Инициализатор
    init(model: String) { 
       self.model = model
    }
}

var car1 = Car(model: "Tesla")
var car2 = car1
car2.model = "BMW"

// ОБРАТИТЕ ВНИМАНИЕ – НАЗВАНИЕ ИЗМЕНИЛОСЬ О ОБЕИХ МАШИН
print(car1.model) // BMW
print(car2.model) // BMW
```

Обе переменные ссылаются на один и тот же объект в памяти. 

Когда мы сказали, что `var car2 = car1` фактически **мы не создали независимый объект**, мы просто сказали, что теперь у нас есть объект `car2` который ссылкается на ту же ячейку в памяти, что и `car1`

**Если мы изменим `car1` или `car2` то другой объект так же притерпит изменения.** Так как они ссылаются на один и тот же адрес в памяти.

Что значит "ссылаются на один и тот же адрес в памяти" ?

Дело в том, что когда мы создаём класс – ему выделяется место в памяти (представьте себе библиотеку, с кучей ячеек) это место в памяти обозначается неким идентификатором (ссылкой), по которой мы можем найти этот созданный объект. 

Скажем, у нас есть созданный `car1` и хранится он по адресу `0x6000001436a0` и теперь, когда мы создадим сколько угодно много объектов на основе `car1` (`var car2 = car1`, `var car3 = car1`...)
**все они будут ссылаться на один и тот же адрес памяти**, а следовательно если мы будем менять любой из них – изменятся все.

Поэтому, когда мы говорим, что например `car2.model = "УАЗ"` это значит, что все остальные `car1`, `car3` и так далее тоже изменят свою модель.

Это очень важный принцип классов, это нужно тщательно усвоить, потому, что на практике, когда перед вами встанет выбор – какой из типов данных использовать, нужно отталкиваться от того, какое поведение мы желаем увидеть, при изменении нашего объекта.

Давайте представим, что пользователь заполняет данные о себе на трёх экранах: 

 - На первом мы спросим его о всех его паспортных данных 
 - На втором о всём имуществе, которое ему принадлежит 
 - На третьем о кличке любимого животного и девичьей фамилии матери 
 - На четвёртом мы оформим на него кредит

**Так как у нас есть куча независимых экранов, на которых мы дополняем информацию о пользователе** – нам удобно выбрать в качестве структуры данных **класс**. 

В таком случае, где бы мы не поменяли данные у нашего реального объекта `user` он всегда будет иметь актуальную информацию. 

Можно долго продолжать описываться это поведение словами, но лучше всего будет проверить это на практике. В конце стать именно этим мы и займёмся.


**Пример со структурой:**

```swift
struct Car {
    var model: String
}

var car1 = Car(model: "Tesla")
var car2 = car1
car2.model = "BMW"

print(car1.model) // Tesla
print(car2.model) // BMW
```

**Здесь `car2` — независимая копия. Изменения в одной не влияют на другую.**

Как вы, я надеюсь, заметили – в случае со структурой объект `car2` полностью независим от `car1`, впрочем как и `car1` полностью независим от `car2`. 

Если мы накопируем миллард каров, никакой из них абсолютно никак не будет влиять на другие.

Дело в том, что структуры имеют полностью иное поведение, они хранятся в совершенно ином месте нежели классы. 

Структуры хранятся в стеке (это прям совсем рядышком с процессором – ооочень быстрое место, процессор за доли наносекунд способен работать со структурами, но размер этого хранилища (Стека) ограничен) 

Классы же хранятся в куче – это своеобразная помойка, она находится дальше от дома (процессора) и гораааздо больше нежели стек (в котором хранятся структуры).

Так как куча (место где хранятся классы) находится дальше от процессора – они медленнее и там ещё бегают крысы – это очень плохо.

Дело в том, что с классами много проблем, особенно когда речь заходит о многопоточном программировании и всё из-за того, что **они делят одну ссылку на всех**. 

Представьте, что в комнате сидят четыре человека и играют в Мафию. В колоде только одна карта Мафии.

Представим, что колода это класс. 

В адекватном мире, мы сначале вручим колоду в руки Пете, затем Маше, потом Васе и наеонец Еблантию. Каждый из них **по очереди** достанет карту – проблем нет.

Но давайте представим, что мы живём в мире, где возможно многопоточное взятие кард из колоды. 

У нас есть четыре потока (Петя, Маша, Вася, Еблантий)  – **теперь они могут выполнять работу одновременно**

Мы не можем предсказать кто из низ возьмёт колоду первым, мы не знаем, кто из них первым вернёт колоду на место – **мы не знаем абсолютно ничего**. **Велик шанс, что они начнут одновременно доставать карту из колоды.**

Если они только посмотрят на неё и не будут менять – проблем нет, но представьте, **что случится если одну и туже колоду, в один и тот же момент будут менять четыре потока** ?

Когда колода вернётся на место – мы вообще без понятия, что с ней случилось, вполне возможно, что у нас окажется две мафии, вполне возможно, что ни одной. 

Проблема в том, что при синхронном программировании – все задачи выполняются шаг за шагом – мы взяли колоду, вытащили карту, отдали колоду следующему.

Но при многопоточном программировании – все могли взять колоду одновременно, одновременно вытащить карту и вернуть на место. 

Если Петя вытащил Мафию, но не успел вернуть колоду на место, Вася тоже емеет шанс вытащить Мафию, так как они пытаются работать с одной колодой одновременно и в итоге наша игра в Мафию сломается – так как будет непредсказуемое количество Мафий (Есть шанс, что вообще ни кто не вытащит карту Мафии или вытащат все – они же делают это одновременно, они не ждут пока кто-то один вытащит, что бы тянуть следующему)

В этом одновременно и проблема и преимущество классов – одни и теже данные мы можем менять от куда угодно, когда угодно.

Если бы наша колода была структурой – то каждый из игроков взял бы полностью независимую копию колоды и мог бы менять её как угодно – это никак бы не отразилось на остальных.

Но ведь тогда теряется смысл игры) – **одну колоду нужно поделить на всех**, а не создать десять копий одной и той же колоды и менять её независимо.

Поэтому колода – должна быть классом, но нам нужно управлять потоками и не разрешать менять колоду, до тех пор, пока с ней работает другой поток.

А что же выбрать для структуры ? 






### 2. Наследование

| Классы | Структуры |
|--------|-----------|
| Поддерживают наследование | **Не поддерживают наследование** |
| Могут переопределять методы (`override`) | **Не могут переопределять** |
| Могут вызывать методы суперкласса (`super`) | **Нет понятия суперкласса** |

**Пример наследования в классе:**

```swift
class Animal {
    func speak() {
        print("...")
    }
}

class Dog: Animal {
    override func speak() {
        print("Гав!")
    }
}
```

**Со структурами такое невозможно.**

### 3. Деинициализаторы

| Классы | Структуры |
|--------|-----------|
| Могут иметь деинициализаторы `deinit` | **Не могут иметь деинициализаторы** |
| `deinit` вызывается перед освобождением объекта | — |

**Пример деинициализатора:**

```swift
class FileHandler {
    var filename: String
    
    init(filename: String) { 
	    self.filename = filename
    }
    
    deinit {
        print("Закрываем файл \(filename)")
    }
}
```

### 4. Изменяемость свойств в методах



**Пример со структурой:**

Если у нас в структуре есть функция, которая менят структуру (присваивает иное значение полю), то такая функция должна быть помечена как `mutating` (`xCode` сам подскажет)

```swift
struct Point {
    var x = 0.0
    var y = 0.0
    
    mutating func moveBy(x deltaX: Double, y deltaY: Double) {
        x += deltaX
        y += deltaY
    }
}

let fixedPoint = Point(x: 3, y: 3)
// fixedPoint.moveBy(x: 2, y: 3) // Ошибка! Нельзя вызвать mutating метод у константы

var movablePoint = Point(x: 3, y: 3)
movablePoint.moveBy(x: 2, y: 3) // OK mutating можно вызвать только у var
```

**В классе такого ограничения нет:**

```swift
class PointClass {
    var x = 0.0
    var y = 0.0
    
    func moveBy(x deltaX: Double, y deltaY: Double) {
        self.x += deltaX
        self.y += deltaY
    }
}

let point = PointClass()
point.moveBy(x: 2, y: 3) // Работает, даже если экземпляр константа
```

### 5. Сравнение на идентичность

| Классы | Структуры |
|--------|-----------|
| Можно сравнивать ссылки с помощью операторов `===` и `!==` (равны ли ссылки на один объект) | Для структур такого оператора нет; обычно сравнивают по свойствам через `==`, если структура соответствует `Equatable` |

**Пример сравнения классов:**

let carA = Car(model: "Tesla")
let carB = carA
let carC = Car(model: "Tesla")

print(carA === carB) // true (одна и та же ссылка)
print(carA === carC) // false (разные объекты)

Структуры не поддерживают `===`, так как у них нет идентичности.

### 6. Время жизни и управление памятью

| Классы | Структуры |
|--------|-----------|
| Живут в куче, управляются через ARC (подсчёт ссылок) | Живут в стеке, уничтожаются при выходе из области видимости |
| Могут создавать циклические ссылки (нужно использовать weak/unowned) | Нет циклических ссылок (не ссылочный тип) |

## Когда выбирать класс, а когда структуру?

`Apple` **рекомендует использовать структуры** по умолчанию, кроме случаев, когда вам действительно нужны возможности классов. Вот типичные ситуации:

### Используйте структуры, если:
- Нужно простое хранилище данных
- Вы хотите, чтобы данные копировались независимо, а не делились на всех по одной ссылке
- Не нужно наследоваться
- Вы работаете с небольшими объёмами данных (например, координаты, размеры)
- Тип представляет собой неизменяемое значение

### Используйте классы, если:
- Нужно наследование (один из ключевых признаков ООП)
- Требуется управление жизненным циклом через `deinit`
- Нужно совместное использование одного экземпляра
- Необходима идентичность объекта (сравнение по ссылке)

## Заключение

Классы и структуры в Swift имеют много общего, но их поведение в памяти и возможности кардинально различаются. 

Понимание этих отличий помогает писать безопасный, эффективный и понятный код. 

Основное правило: **структуры — для значений, классы — для объектов с идентичностью**. Следуя этому принципу, вы избежите многих ошибок и сделаете свой код более производительным.
