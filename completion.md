
# Замыкание (Completion) в SwiftUI

## Что это такое

`Completion` (Обратный вызов / Замыкание и т.д.) — достаточно труднообъяснимый механизм, который позволяет вернуть данные из функции – когда-нибудь потом, либо запросить данные, которые нужны внутри функции извне.

**Я прекрасно понимаю, что вам ничего не понятно – давайте разбирать на примерах.**  

## Загрузка данных из сети

Представим, что нам необходимо загрузить данные из сети, например имя пользователя (но это применимо к абсолютно любым данным).

В зависимости от качества соединения, скорости и размера загружаемых данных, мы можем загрузить их как через час так и через пять секунд. 

**Но как нам вернуть эти данные ? если мы не знаем когда именно они будут загружены ?** 

Мы можем воспользоваться классическим подходом и указать, что наша функция возвращает ` -> String` (Имя пользователя) – вот здесь и кроется главная проблема: 

Дело в том, что код в функции выполняется строчка за строчкой, с каждым тиком процессора мы опускаемся на строчку ниже и функция не может ожидать не пойми чего несколько секунд – **она должна вернуть значение сразу же.**

Если всё упростить, то функция из 10 строчек, должна опускаться на строчку ниже каждую миллисекунду и получается, что к 10ой миллисекунде у нас уже должны быть данные, которые необходимо вернуть из функции.

```swift
func loadUsername() -> String {
0	/// Имитируем загрузку с сервера
1	DispatchQueue.main.asyncAfter(deadline: .now() + 143) {
2	    return "Magdalena"
3	}
4   return ??? // Нам не откуда взять имя так как оно ещё не загрухилось
}
```

В нашем случае – когда мы пытаемся загрузить данные из сети, подобный алгоритм заметно нарушается, ведь **когда мы дойдём до строчки 1, на которой отправим запрос на сервер, мы не будем ждать пока он выполнится, мы сразу же перейдём на строчку 4**, которая является последней строчкой в функции – в ней то мы и должны вернуть ответ от сервера, **но ведь он же ещё не пришёл!**

Вы верно спросите – а **почему от 1ой строчки мы сразу перешли к 4ой ?** 

Дело в том, что подобные тяжеловесные задачи – как загрузка данных с сервера, **выполняются не на главном потоке**, они уводятся на вспомогательные потоки и совершенно не тревожат наш главный поток, позволяя ему спокойно работать.

Если бы мы выполнялись весь наш код только на главном потоке – приложение бы просто зависло на том моменте, пока мы ожидаем ответа с сервера. 

Конкретно в нашем случае мы говорим – `DispatchQueue.main.asyncAfter`, где `async` играет ключевую роль  это значит – **выполни код асинхронно**. 

До этого момента весь наш **код выполнялся синхронно – строчка за строчкой**, но благодаря `async` мы буквально сказали следующее – **слушай, дорогой процессор, давай ты не будешь усложнять себе жизнь и выполни этот код, когда тебе заблагорассудится, как найдёшь свободную минутку.** 

И процессор нас послушался, он запомнил, что ему нужно сделать, но не стал с этим заморачиваться и сразу же перешёл к строчке под номером 4, на которой нам нечего возращать так как ничего не загружено(

В итоге, код внутри блока `async` выполнится не синхронно, а через 143 секунды, когда уже наша функция, а мб и весь класс, а мб и весь экран будут смахнуты пользователем назад, За это время, телефон пользователя могут просто украсть, а самого его пустить на органы. 

Очевидно, что в таком случае мы точно не должны писать `-> String` и пытать что-то вернуть.

Мы должны воспользоваться механизмом `Completion`, о котором и идёт в статье речь.

```swift
func loadData(completion: @escaping (String) -> Void) {
    DispatchQueue.main.asyncAfter(deadline: .now() + 143) {
	    /// Вот так мы вернём значение из функции
        completion("Данные загружены")
    }
}

// Вызов функции с замыканием, где result – то что вернёт нам функция, когда-то, абсолютно не понятно когда.
loadData { result in
	/// Наконец – мы можем получить значение из функции
    print(result)
}
```

Запомните эту запись – `func loadData(completion: @escaping (String) -> Void) {`

Вначале всё как и обычно:

`func loadData` – обозначение функции
`completion` параметр функции (можно назвать как угодно, лишь бы подходил по смыслу)

Далее начинается самое интересное:

`@escaping` – замыкание может убежать (прям как молоко), даже нет смысла это объяснять, **xCode сам подскажет когда это нужно добавить**
`(String) -> Void` – Это замыкание, которое должно вернуть `String` при этом, когда мы будем вызывать нашу функцию – мы не требуем, что бы внутри этого блока что-то возращалось. Мы лишь ожидаем, когда замыкание вернёт нам `result`

```swift
loadData { result in
/// Так как `(String) -> Void` – Void, значит ничего, пустота
/// здесь мы не должны ничего возвращать
}
```

**Это действительно сложно осознать, нужна практика.**

## Разберём на примерах:

Напишем функцию, которая когда-нибудь должна вернуть нам цифру
```swift
func calculate(completion: @escaping (Int) -> String) {
    /// Асинхронный код, результат внутри него нельзя вернуть напрямую
    /// Необходимо использовать замыкания
    DispatchQueue.main.async {
        /// Представим, что цифру 2 – мы получаем от сервера, через API
        /// Получили через пол часа и наконец, вызываем замыкание, в которое передаём полученный результат
        let value = completion(2)
        print(value) // "Два"
        
        let value5 = completion(5)
        print(value5) // "Пять"
    }
    
    /// Мы не можем ничего вернуть так как значения нет
    // return ???
}
```

При вызове данной функции, внутри замыкания (блок `{ in ... }`) мы должны вернуть Строку.

```swift
func hello() {
    /// Вызываем нашу функцию с замыканием
    /// - Parameters:
    ///   - value: значение из замыкание, которое мы когда-нибудь получим (Например цифра 2 и 5)
    calculate { value in
        /// Замыкание требует, что бы при вызове мы вернули строку
        /// Давайте вернём буквенное обозначение цифры
        switch value {
        case 2:
            return "Два"
        case 5:
            return "Пять"
        default:
            return "Неизвестное значение"
        }
    }
}
```

## В модификаторах SwiftUI

 Есть куча хорошо известных нам замыканий, к примеру:

```swift
.onAppear / onDisappear:
.onAppear { startTracking() }
.onDisappear { stopTracking() }
.refreshable { await refreshData() }
```

Код в блоке `onAppear` должен выполниться в тот момент, как `View` будет показана на экране, при этом за вызов этого замыкания отвечает лично `Стив Джобс` ведь у нас нет никаких данных о том, когда то или иное `View` может появиться перед зелёными глазами пользователя. 

Выполняя код `onAppear` или `.onDisappear` мы как бы подписываемся на событие показа / скрытия экрана, а система сама нас об этом предупредит, когда посчитает нужным.
